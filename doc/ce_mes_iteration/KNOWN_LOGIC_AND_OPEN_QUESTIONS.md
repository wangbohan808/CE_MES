## 2. 协议表格（CSV）中已摘录的帧结构

约定（来自表头列）：**协议头 `A5 5A` → 长度 → 设备 → 命令 → 数据 → `sum`（校验）**。

| 名称 | 设备 | 命令 | 长度（表中） | 数据要点 |
|------|------|------|--------------|----------|
| 开始测试 `>>` | `50` | `66` | `03` | 数据区含 `00` 等 |
| 阈值上传 `>>` | `50` | `68` | 变长 `xx` | 回充码、基站版本、频率、尘袋/充电/LED/集尘等阈值类字段 |
| 扫码/MES 应答 `<<` | `50` | `0x57` / `0x58` | `02` | `57`：扫码枪通过继续；`58`：MES 返回错误（表意，与实现需一致） |
| 实时数据 `>>` | `50` | `77` | `24` | 步骤、回充码、版本、频率、尘袋、充电、LED、集尘（注明单位 10Pa）等 |
| 结束 `>>` | `50` | `88` | `01` | 数据 `01/02/03`：成功 / 失败 / 结束（表意） |

**CSV 未写清**：`sum` 算法与覆盖范围；各 `XX`/`HH`/`LL` 字段精确字节宽度与字节序；单独出现的 `89` 与孤立 `03` 行与 `88` 的关系。

## 3. 业务方向与流程（来自需求描述，待与固件最终对齐）

### 3.1 方向约定

- **`>>`**：RV30 治具 → 本软件（上位机）：同步数据，需**解析**。
- **`<<`**：本软件 → 治具：**反馈/应答**，用于扫码门闸或**推动测试进入下一步**（推动帧的具体格式见「待澄清」）。

### 3.2 流程理解（文字状态机）

1. **阈值上传（`68` 等）**  
   治具侧会上传阈值类数据；本侧策略为：**解析后写入 `config.yaml` 作为参考**，供后续与实时数据比对、判 Pass/NG。

2. **开始测试（`66`）**  
   治具发送「开始测试」完整帧；本侧成功组帧并解析后，根据**扫码枪是否通过**发送 `<<`（`0x57` / `0x58`）。**通过**则进入正式测试流程。

3. **测试中（`77`）**  
   治具按工步**在同一类实时数据帧内逐步填充内容**；需求侧强调：**帧长度/布局不变**，随流程「逐步丰富」有效数据。过程中本侧需**向治具反馈**以**推动下一步**。

4. **结束（`88`）**  
   若顺利至最后一步且无报错，治具发送结束帧，本轮流程结束。

## 4. 代码中已确认的函数职责（`test_tool/test.py`）

### 4.1 `test_serial_rx_data_handle()`（约 431–450 行）

- **若** `test_serial.test_rx_q` 非空：取出本次收到的**字节列表**，对每个字节调用 `test_rx_data_handle(hex_dat)`，并可选写日志。
- **否则若** `rx_sn_cmd_q` 非空：取出模拟命令，使用 `load_cfg.dev` 与队列中的 `cmd`、`msg`，**直接**调用 `test_cmd_handle(dev, cmd, data)`（**不经过**字节流组帧），用于调试/模拟。
- **分支优先级**：`if` 优先于 `elif`——只要接收队列有数据，**同一轮不会**处理 `rx_sn_cmd_q`。
- **两队列皆空**：函数不执行有效业务。

### 4.2 `test_rx_data_handle(hex_dat)`（约 461–511 行）

- **按单字节**流入，维护全局缓冲 `pack_data` 等，完成 `A5 5A` 同步、长度 `pack_data_len`、设备 `check_dev`、命令 `check_cmd`、数据区 `check_data` 的提取。
- **校验**：对长度、设备、命令及数据区字节做 `check_sum` 累加，最后一字节与 `check_sum % 256` 比较；通过则调用 `test_cmd_handle(check_dev, check_cmd, check_data)` 并清空缓冲；失败则清空并打印错误。

### 4.3 `test_cmd_handle(dev, cmd, dat)`（约 522–554 行）

- `dat` 长度 `< 1`：视为数据异常，UI 提示「治具数据异常」并返回。
- `dev` 与 `load_cfg.dev` 不一致：提示「治具类型不匹配」，**不进入**各 `*_mode` 分支（注意：仍会 `print`，与 `check_cfg_dev` 等其它路径的关系若需统一行为可再查）。
- 否则按 `dev` 分发到不同治具逻辑；**设备 `50`** 当前走向：`hw1_bastation_finished_product_mode_FX(dev, cmd, dat)`（命名含拼写 `bastation`，搜索时注意）。

### 4.4 发送辅助（同文件约 556–574 行）

- `ser_send_cmd(dev, cmd)`：短帧，`0xA5 0x5A 0x02 dev cmd ck_sum`，其中 `ck_sum = (0x02 + dev + cmd) % 256`。
- `ser_send_data(dev, cmd, data)`：变长，`check_sum` 经 `tool.check_sum([0x02 + data_len, dev, cmd] + data)`。

**注意**：接收侧 `test_rx_data_handle` 的累加方式与 `ser_send_cmd` / `ser_send_data` 是否完全一致，应在联调时逐命令核对（见待澄清）。

## 5. 待澄清问题清单（优先级不分先后）

1. **`<<`「推动下一步」的帧格式**  
   是否仍为 `A5 5A` + 长度 + `50` + **专用命令字** + `sum`？命令是否仅有 `57/58`，还是另有 ACK/步号？每步是否只允许发一次？

2. **`0x77`「长度不变、逐步丰富」的精确定义**  
   未生效字节是 `0x00`、`0xFF` 还是其它占位？是否存在「有效位」或「步骤号」与本软件发 `<<` 的严格对应表？

3. **`0x68` 与 `0x66` 的时序**  
   是否每次会话必现 `68` 再 `66`？若未收到 `68` 是否禁止进入测试？

4. **`0x58` 语义**  
   是否仅表示 MES 错误，还是扫码失败也走 `0x58`？发送后治具是否还会发 `0x77`/`0x88`？

5. **长度字段 `pack_data_len` 的语义**  
   表示「设备+命令+数据」总字节数，还是含其它范围？需与治具固件、现有 `ser_send_*` 完全一致。

6. **`sum` 算法**  
   接收路径当前为对「长度+设备+命令+数据字节」累加后 `%256` 与末字节比较；是否与治具对每一类帧（尤其 `0x24` 长数据帧）完全一致？

7. **`0x88` / `0x89` 与结束语义**  
   CSV 中 `89` 与孤立 `03` 是否启用？失败路径下是否仍可能收到 `0x88` 及 `01/02/03` 的组合？

8. **超时与重试**  
   某步未收到下一帧或未收到应答时的超时、重试、UI 提示策略。

9. **`test_cmd_handle` 在 `dev` 不匹配时的行为**  
   是否应统一为直接 return，避免与其它检查重复或行为不一致。

10. **`rx_sn_cmd_q` 模拟路径**  
    `data` 类型需与真实 `check_data`（`list` of int）一致，否则分支内可能异常；生产环境是否应禁用或加开关。
