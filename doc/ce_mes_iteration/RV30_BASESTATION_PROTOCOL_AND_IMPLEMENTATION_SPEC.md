# RV30 基站成品治具 ↔ 上位机（CE_MES）— 协议与实现规格（移交稿）

> **目的**：让后续 AI 或工程师在不依赖口头上下文的情况下，直接实现或评审 `device_type == 50`（及同类新协议）逻辑。  
> **权威**：字段布局与字节含义以 **`doc/通讯协议.png`** 为准；`doc/MES协议.csv` 可作辅助。  
> **收发实现**：必须与现有 `test_rx_data_handle` / `ser_send_cmd` / `ser_send_data` 一致（见 §2）。

| 用途 | 路径 |

## 2. 帧格式（与现有代码完全一致）

### 2.1 接收（`test_rx_data_handle`）

1. 同步字：`0xA5`、`0x5A`。  
2. 第 3 字节 **`pack_data_len`**：**从设备字节到数据区最后一个字节**的总字节数，即 **设备(1) + 命令(1) + 数据区(n)**，故 **`n = pack_data_len - 2`**，即传入 `test_cmd_handle` 的 **`len(dat)`**。  
3. 最后一字节为 **SUM**；校验为从「长度字节」起至「数据区末字节」累加后 **`% 256`** 与 SUM 相等。

### 2.2 发送

- **`ser_send_cmd`**：`A5 5A 02 dev cmd sum`（无数据区）。  
- **`ser_send_data`**：第 3 字节为 **`0x02 + len(data)`**，等于 **2 + 数据区长度**（与接收侧「长度」语义一致）；后跟 `dev`、`cmd`、`data…`、`sum`（`tool.check_sum`）。

**禁止**为新治具另写一套长度或校验算法，否则与固件不一致。

## 3. 方向与命令语义

| 方向 | 命令 | 含义 | 上位机动作 |
|------|------|------|------------|
| >> | `0x66` | 开始测试 | 清报表、清扫码队列、`check_sn_enable=True`、复位 UI；**不**发 `0x67`；仅通过后续 **`0x57`/`0x58`** 放行或拒绝。 |
| >> | `0x68` | 阈值上传 | 解析并缓存（可选写回 YAML）；**长度见 §8 草稿**，后续人工按 PNG 修正。 |
| >> | `0x77` | 实时数据（约 500ms） | 只更新内部状态 / 判阈 / UI；**不对治具回任何应答帧**。 |
| >> | `0x88` | 结束 | 进入**会话收尾**：结合 §6 规则判定 MES；**不**要求与 `0x88` 首字节 `01/02/03` 做严格 OK/NG 映射。 |
| << | `0x57` | 通过 | 数据区 = 扫码 SN 的 **UTF-8 字节**（与现 `barcode_check_process` 一致）。 |
| << | `0x58` | 拒绝/错误 | 同上；**设备号**指帧内 **设备字节**（如 `0x50`），**不**在数据区额外拼设备号字符串。 |
| << | `0x89` | 异常结束 | 数据区含 **`0x03`**，即 `ser_send_data(dev, 0x89, [0x03])`。 |

## 4. 正常业务流程（时序）

1. **可选**：治具 `0x68` → 上位机解析阈值（§8）。  
2. 治具 `0x66` → 上位机：`mes_run.clear_report`、`tool.clear_queue(barcode_q)`、`check_sn_enable=True`、`reset_ui`、提示扫码；**不发串口应答**。  
3. 扫码 → `barcode_q` → **`barcode_check_process()`**（在主循环中**先于**串口处理）：编码规则 + `mes_run.check_sn_is_ok` → **`0x57`** 或 **`0x58`**。  
4. 治具周期发 **`0x77`** → 解析步骤与各测量项；与 **`config.yaml` 中预先填写的阈值** 比对（§5）；按步骤变化策略刷新 UI（步骤表仅作**何时刷新**的参考，非刚性协议）。  
5. 治具 **`0x88`** → 上位机执行 **§6 会话收尾与 MES**，**不**依赖「仅因收到 `0x88` 才第一次算 OK/NG」的单一映射。

---

## 5. 实时数据与阈值来源

- **主来源**：**`config.yaml` 中预先填写**的阈值/上下限/目标版本等；实现时从 `load_config()` 已加载的 `load_cfg` 或解析后的结构读取。  
- **`0x68`**：可作为动态阈值或覆盖；与 YAML 的关系由产品定义（例如仅启动时参考、或覆盖内存中的比较用副本）。**比对逻辑必须显式包含「以 YAML 为主」这一路径**，避免只依赖 `0x68` 而未落盘导致重启不一致。  
- **`0x77`**：字段偏移与单位（如集尘 **10Pa**）以 **`通讯协议.png`** 为准编写解析表。

## 6. MES 与结束语义（用户已定稿）

### 6.1 `0x88` 数据 `01/02/03`

- **没有**与 MES **OK/NG** 的**严格一一映射**；不要写 `if dat[0]==0x01: MES_OK` 这类作为唯一真理。

### 6.2 MES OK/NG 如何判定

- **任一方结束会话后**（治具发来结束相关帧，或上位机已走异常收尾），由**上位机根据本轮累积结果**判断是否 **「正常结束」**，再决定 MES 报 OK 还是 NG。  
- 用户业务描述：**正常结束时，结束态双方语义上只使用 `03`**（与协议中「结束/异常」类数据字节的约定一致；具体指 `0x88` 数据区、`0x89` 载荷等以 PNG 为准实现）。实现上应：  
  - 维护本轮 **各工步结果**、**是否已发 `0x89`**、**是否收到 `0x88`** 等标志；  
  - 在**收尾函数**中统一调用 `mes_run.add_report` / `send_report`（或项目现有等价 API），**用综合结果**写 OK/NG，而不是单字节映射表。

### 6.3 「仅发 `0x89`」的异常路径与 `0x88`

- 在**实时数据阶段**仅发 **`0x89` + `0x03`** 终止异常时：**不需要等待 `0x88`** 再上报 MES。  
- 应在发送 `0x89`（及必要的防抖）后**尽快**走 MES 失败/异常上报与 UI 终态，避免死等 `0x88`。

### 6.4 开始测试阶段异常（门闸前）

- 顺序：**先发 `0x58`**（设备字节为配置 `dev`，数据区与现失败路径一致），**再发 `0x89` + `0x03`**。  
- MES：按失败处理；同样**不必**等待 `0x88`。

---

## 7. 异常与防抖（与先前结论一致）

| 阶段 | `0x58` | `0x89` + `0x03` | 等待 `0x88` 再 MES |
|------|--------|------------------|---------------------|
| 开始测试 / 门闸 | 视需要先发 | 随后发 | **否** |
| 实时数据 | 不发 | 仅发 | **否** |

对 **`0x89`** 与 MES 上报做 **once** 防抖，避免每个 `0x77` 周期重复发送。

---

## 8. `0x68` 阈值帧长度（草稿，待你按 PNG 修改）

以下依据 **`MES协议.csv`** 与 **`通讯协议.png`** 结构估算**数据区**字节数（**不含**设备、命令；含数据区内全部字段的连续字节），**仅作开发起点**：

数据区大致顺序（与表一致）：  
回充码左/左中/右中/右（4） + 基站版本 4 字节 + 频率（1，若固件为多字节则以 PNG 为准）+ 尘袋（1）+ 充电 min HH/LL + max HH/LL（4）+ LED（1）+ 集尘 min HH/LL + max HH/LL（4）。

**粗算数据区 ≈ 4 + 4 + 1 + 1 + 4 + 1 + 4 = 19 字节**（频率/尘袋/LED 宽度以 PNG 为准可增减）。  
**帧内长度字节** = **2 + 数据区长度**（设备+命令+数据）。  

**另一 AI**：解析 `0x68` 时使用**命名常量 + 注释「待联调修订」**；长度校验可先用 `len(dat) >= 最小期望` 软校验并打日志，待用户改 PNG/固件后替换为精确长度。

## 9. 代码书写思路（实现清单）

1. **隔离变更**：仅改 `dev == 50`（或配置约定的新 `device_type`），保留其他 `dev` 的 `0x67` 旧逻辑不动。  
2. **状态机**（建议枚举或常量）：如 `IDLE` → `WAIT_SN`（已收 `0x66`）→ `RUNNING_77`（已发 `0x57`）→ `FINISHED` / `ABORTED`；异常从 `WAIT_SN` 或 `RUNNING_77` 进入 `ABORTED` 并触发 MES。  
3. **`0x66`**：不发送 `0x67`；只做清队列、使能扫码、记 `test_start_time`、UI。  
4. **`barcode_check_process`**：失败分支接 **`0x58` → `0x89` + `0x03`**；成功后 **`0x57`** 进入 `RUNNING_77`；**不**等待 `0x88` 再报 MES。  
5. **`0x68`**：按 §8 草稿偏移解析到内存结构；写不写回 `config.yaml` 由产品决定，建议与现有 `dust_th` 或新命名空间对齐。  
6. **`0x77`**：按 PNG 建下标表；保存 `last_step`；步骤表仅控制**何时**做阈值比较与 UI 刷新；阈值**以 `config.yaml` 为主**与实时字段比较；超阈：**仅** `ser_send_data(dev, 0x89, [0x03])`，然后 **立即 MES 异常收尾**（不等待 `0x88`）。  
7. **`0x88`**：调用统一 **`finalize_session()`**：根据累积结果 + 是否正常结束（含用户「双方结束态 03」约定）设置 MES，**不**单独依赖 `dat[0]` 与 OK/NG 严格映射。  
8. **线程/UI**：串口与 `wx` 更新继续用 **`wx.CallAfter`**，与现工程一致。  
9. **联调**：打印完整 hex 与 `pack_data_len`、`len(dat)` 对照 §2。
